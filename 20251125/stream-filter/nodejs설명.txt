프로세스 vs 스레드

	프로세스: 프로그램을 실행했을때 한자리를 차지하는 애-OS 위에 프로세스가 여럿있음-멀티테스킹(멀티 프로세싱)
		-옛날에는 프로그램을 실행하면 프로그램 하나만 돌아갔음. 다른거 실행하려면 기존의 실행 프로그램을 종료하고 실행해야했음.
		-그런 불편한 점을 해결하기 위해서 멀티 프로세싱이 등장함.
		-여러개의 프로그램이 어떻게 돌아가게 하는 것인가?
			-timeslice: 시간대 별로 각각 다른 프로세스가 os를 점유하게 하는 것
		-context switching: cpu의 상태를 저장하고 복원하는 것,
			-cpu - register
			-cpu의 상태를 복사 후 메모리에 저장 -> 다음 프로세스로 넘어갔을때 -> 이 상태를 저장함 (반복)
			-오버헤드 발생
			- OS가 context switching이 일어날 때는 일을 하지 못함.
				-그래서 멀티코어를 사용하게됨.
					-멀티코어: cpu안에 cpu가 여러개임.
		stack <-> heap -data - code

	스레드: 프로세스 위에 스레드가 여러개 돌아가는 것
		- 하나의 프로세스에서 여러개의 스레드를 돌릴수 있게 하도록 하자는 취지에서 생겨남.
		- 작동 구조는 프로세스와 동일함.
			-단, 프로세스의 context switching보다 훨씬 작은 단위로 돌아감.

		- stack을 여러개 만들어서 사용함.
		- 스레드를 사용하는 것이 어려운 이유:
			-stack을 여러개 사용할 수 있지만 heap을 공유함.
				-heap을 공유하기 때문에 잘못 사용하거나 실수하면 다른 스레드에 영향을 줄 수가 있음.
			-멀티 스레드의 경우:
				- 코드 자체는 문제가 없는데 특정 조건에서 에러가 나거나 그런 경우가 생기기도함.

		-자바의 경우 tomcat에서 멀티스레드를 사용함. (자바는 알아서 관리함.)
			- 사용자와 서버간에 멀티스레드를 사용해서 여러처리를 하도록 함.
	
stack: 선입 후출- 들어오는 순서대로 쌓이기에 꺼낼때 최근 순서대로 꺼냄 ex) 1234저장 -> 4321순으로 꺼냄
	순서저장-> 파라미터 저장 -> 파라미터 값을 꺼냄.
heap: 다이나믹 메모리
	-c언어는 책임과 권한이 있음.
	-자바는 그런게 없음.

위와 같은 얘기를 한 이유: node는 싱글 스레드임.
	node를 사용하면 빠른 이유:
		싱글 스레드를 사용
			-여러 고객을 멀티 스레드로 처리해도 처리가 힘든데 어떻게 싱글 스레드로 처리가 가능한가?
				- 순번 처리
		비동기 처리
			왜 비동기인가? js는 동작 방식 자체가 비동기임.
	node 개발자의 포커스
		-javascript에 집중
			-js의 단점: 웹 브라우저가 있어야만 동작됨.
		
			-이 단점을 보완하기 위해 함.
			-v8: 크롬에 있는 js를 처리하기 위한 프로그램
				-엔진 오픈 소스 공개함.
				-이 엔진을 modify하게 꺼내서 통신이나 이런 필요한 기능들을 붙여서 만든 것
			-js는 다른 언어와 달리 그 자체가 비동기 방식임.
				-이벤트 방식으로 처리함.

	백엔드에서 nodeJS를 사용하는 이유:
		-속도가 빠름
		-풀스택:
			- 하나의 언어로 모든 작업을 할 수 있는것
			- 폴리글랏: 상황에 따라서 기술이나 언어를 최적화 된 것을 사용하는 것
		-기능은 적고 요청은 많은 경우에 사용하는 것이 바람직하다.
			

	동기 처리와 비동기 처리의 차이:

	restful하게 언어의 종류와 상관없이 처리되도록 하는게 현재의 방식임.
		-옛날처럼 언어에 따라서 서버를 새로 살 필요없이 컨테이너 단위로 돌릴 수 있음.
