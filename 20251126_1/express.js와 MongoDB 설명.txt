2. node의 DB연결
	- 3V (빅데이터의 구조)
		- 정규화 ( ex) 원본 데이터에서 불필요한 것을 제외하고 꼭 필요한 것만 저장함.)
		- 속도
			-최대한 빨리 분석해서 결과가 나와야함
		- 다양성
			- 적합성과 같은 것을 알려면 정규화된 데이터로는 알수가 없음.
			- 어떤 데이터가 필요할 지 필요하지 않은지 알 수가 없음.
			- 컬럼으로 미리 정해두면 버려지는 데이터가 생기기 때문에 컬럼 개념이 아닌 통으로 저장함.
			- 데이터를 구분하기 위해서 pk값만 유지하고 데이터의 형식은 구애받지 않음.
		
	- 동기형 데이터베이스인 mysql과 oracle같은 애들이랑은 안어울림
		- 성능 저하
		- 정규화
		
	- nosql이 잘어울림 (Not Only SQL - 빅데이터)
		- 특히 mongoDB
			왜? 버라이어티한 데이터 방식을 mongoDB만큼 처리할 수 있는 DB가 없음
				- 데이터를 JSON형태로 저장함 (문자열 형식)
				- BSON (JSON형태를 바이너리 형태로 변환해서 저장함.)
				- 객체 형태로 저장하기 때문에 변환 작업이 필요가 없어서 빠름.
				- 컬럼이 없기 때문에 데이터 별로 안의 구성 내용이 다를수 있음 ( EX) 테이블 A - 컬럼 3개, 테이블 A - 컬럼 5개, 테이블 A - 컬럼 10개)
				- 데이터에 function을 주면 데이터에 function을 돌려서 해당하는 데이터만 가져옴.
				- Collection =RDB의 테이블 같은 개념 = Document의 집합체
				- Document = JSON을 바이너리화 해서 SQL대신 읽고 쓰고 한다.
				- 분산하기 좋음 (데이터가 모여있을 필요가 없음.) -> 저장을 나눠서 할수있음 -> 큰 머신이 필요없고 작은 머신에 나눠서 일을 시킬 수 있음.
